## 2. 当前与 Node 演示版强绑定的脚手架盘点

### 2.1 代码与目录绑定

1. `app/server.js` 承担 API 路由、静态资源托管、会话内存状态管理，后端能力全部与 Node 运行时绑定。
2. `app/public/` 由 Node 静态文件服务直接暴露，默认依赖同仓单体运行模式。
3. `scripts/smoke.mjs` 依赖 Node 运行时执行，作为主链路烟测入口。

### 2.2 构建与命令绑定

1. 根目录 `package.json` 仅定义 `dev/start/smoke` 三个 Node 命令，未提供 Go 构建入口。
2. 仓库启动说明（`README.md`）当前仅覆盖 Node 服务启动路径。

### 2.3 数据与证据绑定

1. `data/demo-fixtures.json` 目前被 Node 服务直接读取，缺少 Go 服务约定的配置装载边界。
2. `evidence/2026-03-27/` 的生成流程与 `scripts/smoke.mjs` 绑定，尚未声明 Go 运行模式下的兼容策略。

## 3. Go 服务目录、构建命令与 README 组织建议

## 3.1 目录建议（保持“演示可运行 + Go 并行落地”）

建议新增如下 Go 目录，不立即删除现有 Node 路径：

```text
cmd/interview-api/main.go
internal/app/                # 用例编排（会话、题目、答案、结果）
internal/domain/             # 领域模型与规则
internal/transport/http/     # 路由与 Handler
internal/infrastructure/     # fixture 读取、日志、存储抽象
pkg/observability/           # requestId/traceId 中间件
configs/                     # 本地配置样例
```

取舍说明：

- 保留 Node 路径作为演示兜底，避免一次性切换导致当日演示中断；
- Go 路径按 `cmd + internal + pkg` 拆分，后续可平滑扩展到 Kubernetes/Operator 场景。

## 3.2 构建命令与脚本命名建议

建议用 `Makefile` 统一入口，并保留 npm 命令做兼容壳层（短期不破坏既有习惯）：

- `make dev-go`：本地启动 Go 服务；
- `make test-go`：运行 Go 单测；
- `make smoke-go`：执行主链路烟测；
- `make dev-node-legacy`：启动 Node 演示版（过渡期保留）；
- `make smoke-node-legacy`：执行 Node 版烟测（回归对照）。

脚本命名建议：

- `scripts/smoke-go.sh`（或 `scripts/smoke-go.mjs`）；
- `scripts/smoke-node-legacy.mjs`（由现有 `smoke.mjs` 重命名而来）；
- `scripts/dev-go.sh`、`scripts/dev-node-legacy.sh`。

## 3.3 README 组织建议

根 `README.md` 改为“双路径声明”：

1. **当前稳定演示路径**（Node legacy）；
2. **目标正式路径**（Go primary）；
3. 统一的命令索引（优先展示 `make`）；
4. 文档导航（研发/架构/运维）指向 `工作目录/`。

并新增：

- `docs/runtime/node-legacy.md`：说明 Node 保留边界与退役条件；
- `docs/runtime/go-primary.md`：说明 Go 启动、测试、烟测流程。

## 4. 烟测脚本、样例数据与证据目录兼容方案

### 4.1 烟测脚本兼容

核心原则：**不改接口契约，只替换被测服务实现**。

- 继续使用同一条 API 主链路（`interview-sessions -> questions -> answers -> finalize -> result-summary`）；
- 烟测脚本参数化 `BASE_URL` 与 `CAPTURE_DIR`，使 Node/Go 共用一套验证步骤；
- 增加 `RUNTIME=node|go` 维度用于证据标识，避免覆盖。

### 4.2 样例数据兼容

- 将 `data/demo-fixtures.json` 定义为跨运行时共享的 canonical fixture；
- Go 服务通过 `configs/` 或环境变量加载该 fixture；
- 若未来拆分多场景，按 `data/fixtures/*.json` 管理并保持字段契约稳定。

### 4.3 证据目录兼容

建议证据目录从“仅日期”扩展为“日期 + 运行时”：

```text
evidence/2026-03-27/node/...
evidence/2026-03-27/go/...
```

这样可并行对比 Node/Go 响应与 `requestId` 行为，便于回归与 CTO 评审。

## 5. 最小迁移顺序（不影响当前可演示版本）

### Phase 0：冻结 Node 演示版能力边界（当日可执行）

1. 仅允许修复阻断演示的缺陷，不新增业务能力；
2. 标记 Node 为 `legacy demo runtime`。

### Phase 1：引入 Go skeleton（不切流量）

1. 新建 `cmd/`、`internal/`、`pkg/` 目录与最小启动入口；
2. 先跑通 `healthz` 与 `requestId` 中间件；
3. 保持前端与 Node 路径不变。

### Phase 2：按接口契约补齐 Go 主链路

1. 按现有 canonical API 补齐五个核心接口；
2. 与现有 smoke 用例对齐，确保返回结构兼容。

### Phase 3：双运行时并行烟测

1. 同日生成 Node/Go 两份证据；
2. 对比状态流转、错误码、`requestId` 行为差异并收敛。

### Phase 4：切换默认开发入口到 Go

1. `README` 默认命令切换为 Go；
2. Node 保留一段观察期作为回滚兜底；
3. 观察期结束后计划移除 Node 服务主逻辑。

## 6. 建议清单（可直接执行）

1. 新增 `Makefile` 并定义 `dev-go/test-go/smoke-go` 与 `dev-node-legacy/smoke-node-legacy`；
2. 将现有 `scripts/smoke.mjs` 重命名为 `scripts/smoke-node-legacy.mjs`，并补 Go 版本烟测入口；
3. 增加 `docs/runtime/go-primary.md` 与 `docs/runtime/node-legacy.md`；
4. 调整证据目录结构为 `evidence/<date>/<runtime>/`；
5. 在父任务 [CLOA-61](/CLOA/issues/CLOA-61) 下安排 Go skeleton、烟测兼容、README 重构三项并行子任务。

## 7. 需要 CTO 决策的取舍点

1. Go 框架选择（标准库 `net/http` 还是引入 Web 框架）；
2. 烟测工具继续用 Node 脚本还是改为 Go 测试二进制；
3. Node 观察期时长（建议 1~2 个迭代）与最终退役窗口。
