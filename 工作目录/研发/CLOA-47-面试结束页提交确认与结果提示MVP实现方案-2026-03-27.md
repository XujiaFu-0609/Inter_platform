## 2. 目标与范围

### 2.1 目标

完成候选人面试结束页与提交流程闭环，保证“确认提交 -> 处理中 -> 成功/失败反馈”链路可见、可重试、可回溯，覆盖：

1. 提交确认与防重复提交；
2. 提交后处理中/成功/失败三态提示；
3. 失败场景重试与支持指引；
4. 结束页展示会话基础信息（会话号、时间、题目数）。

### 2.2 本次 MVP 范围

1. 在 `/session/:id/finish` 提供结束页框架与会话信息卡。
2. 提供“确认提交”弹层与二次确认交互。
3. 调用提交接口并展示 `processing/success/failed` 三态反馈。
4. 支持失败后的同页重试，不清空关键会话上下文。
5. 支持提交成功后轮询结果摘要并展示“结果已生成/处理中”提示。
6. 提供“联系支持”兜底路径与错误 requestId 展示。

### 2.3 不在本次范围

1. 面评详情与维度拆解可视化（由 [CLOA-52](/CLOA/issues/CLOA-52) 后续承接）。
2. 结束页问题标签归类展示（Week 2 Should 项）。
3. 候选人历史场次聚合入口（后续版本补充）。

## 3. 页面结构与交互

### 3.1 页面结构

1. 顶部状态区：当前提交状态标签（待提交/处理中/成功/失败）。
2. 会话信息卡：`sessionId`、开始时间、结束时间、题目总数、已作答题数。
3. 结果提示区：处理中说明、成功说明、失败原因与操作建议。
4. 操作区：确认提交、再次提交、返回首页、联系支持。
5. 辅助区：提交时间、最近一次请求 ID、帮助文案链接。

### 3.2 关键交互路径

1. 候选人在面试页点击“提交本场回答”后进入结束页并弹出确认框。
2. 确认后进入 `processing`，按钮禁用并展示加载反馈。
3. 提交成功进入 `success`，展示“已提交成功”及后续结果说明。
4. 提交失败进入 `failed`，展示错误信息并允许重试提交。
5. 若结果摘要暂未可用，结束页保持“结果处理中”并轮询状态。

## 4. 前端状态模型

### 4.1 页面级状态

- `ready_to_submit`：可确认提交。
- `submitting`：提交请求进行中（防重复点击）。
- `submit_success`：提交已成功受理。
- `submit_failed`：提交失败，可重试。
- `result_pending`：提交成功但结果摘要尚未可读。
- `result_ready`：结果摘要可展示。

### 4.2 状态流转规则

1. `ready_to_submit -> submitting -> submit_success`
2. `submitting -> submit_failed -> submitting`（重试）
3. `submit_success -> result_pending -> result_ready`
4. `submit_success -> result_ready`（结果已可直接读取）

### 4.3 防重复提交策略

1. 首次点击确认后立刻设置本地 `inflightLock` 并禁用提交按钮。
2. 每次提交请求携带 `idempotencyKey`（基于 `sessionId + timestamp` 生成）。
3. 页面刷新后若检测到未完成提交，恢复为 `submitting` 或 `result_pending`。
4. 收到 `FINALIZATION_ALREADY_ACCEPTED` 时按成功回放处理，不提示失败。

## 5. 接口协作约定（对齐 [CLOA-44](/CLOA/issues/CLOA-44)）

### 5.1 期望接口

1. `POST /api/v1/interview-sessions/{sessionId}/finalize`
   - 请求：`finalizedAt`、`idempotencyKey`
   - 返回：`status=submitted`、`submittedAt`
2. `GET /api/v1/interview-sessions/{sessionId}`
   - 返回：`status`、`timeline`、`resultAvailable`
3. `GET /api/v1/interview-sessions/{sessionId}/result-summary`
   - 返回：`status`、`summary`、`dimensionScores[]`、`hireRecommendation`

### 5.2 前端请求策略

1. 提交采用单飞请求（同一时刻仅允许一个 `finalize` 在途）。
2. 提交成功后每 3 秒轮询一次会话状态与结果可用标记，最多轮询 10 次。
3. 轮询期间页面不可见时降频到每 8 秒。
4. 超过轮询上限仍未产出结果时，停留 `result_pending` 并提示稍后查看。

### 5.3 错误码映射策略

1. `SESSION_STATE_CONFLICT`：提示“会话状态已变化，正在刷新最新状态”。
2. `FINALIZATION_ALREADY_ACCEPTED`：按成功态处理并进入结果轮询。
3. `SESSION_NOT_FOUND`：提示链接失效，提供返回入口与支持联系方式。
4. `PERMISSION_DENIED`：提示无权限提交，建议联系招聘支持。

## 6. 异常与可恢复策略

### 6.1 提交失败

1. 保留会话信息卡与最近一次输入上下文。
2. 展示“再次提交”按钮，不需要候选人重新作答。
3. 失败提示区保留 `requestId` 便于排障。

### 6.2 网络中断

1. 首次失败自动重试 1 次。
2. 连续失败后切换 `submit_failed` 并提示手动重试。
3. 网络恢复后可直接复用上一次 `idempotencyKey` 重放请求。

### 6.3 页面刷新恢复

1. 刷新时读取本地 `finish:{sessionId}:submitState`。
2. 若本地状态为 `submitting/result_pending`，自动恢复轮询链路。
3. 结果确认可读后立即清理本地提交锁与缓存。

## 7. 埋点设计（MVP）

### 7.1 事件列表

1. `finish_submit_confirmed`
   - 触发：确认提交按钮点击
   - 字段：`sessionId`、`answeredCount`、`totalCount`
2. `finish_submit_result`
   - 触发：提交返回
   - 字段：`sessionId`、`result(success|failed)`、`errorCode?`、`requestId?`
3. `finish_result_polling_state`
   - 触发：轮询状态变化
   - 字段：`sessionId`、`status`、`pollingRound`
4. `finish_retry_clicked`
   - 触发：失败后点击重试
   - 字段：`sessionId`、`lastErrorCode`

### 7.2 埋点质量要求

1. 失败类事件必须带 `errorCode` 与 `requestId`（若后端返回）。
2. 所有事件统一带 `sessionId` 与页面时间戳。
3. 埋点失败不得阻塞提交流程与页面跳转。

## 8. 验收清单

1. 候选人可在结束页进行二次确认提交，且不可重复点击触发并发提交。
2. 提交后可见 `处理中/成功/失败` 三态反馈，文案与动作明确。
3. 失败场景可同页重试或联系支持，不丢失会话基础信息。
4. 页面可稳定展示 `sessionId`、时间信息、题目总数与已作答数。
5. 提交成功后可拉取并展示结果摘要，或提示结果处理中。

## 9. 实施拆分与工时（1.5 人日）

1. 0.5 人日：结束页结构、会话信息卡、确认弹层。
2. 0.5 人日：提交状态机、幂等提交流程、防重复提交。
3. 0.25 人日：异常兜底、重试恢复、本地状态持久化。
4. 0.25 人日：埋点接入、联调走查与验收回归。

## 10. 联调用例（最小集合）

1. 正常提交：确认提交后成功进入 `submit_success`，可看到结果摘要或等待提示。
2. 重复提交：短时间重复点击仅产生一次后端提交请求。
3. 提交失败重试：首次失败后点击重试成功，状态从 `submit_failed -> submit_success`。
4. 刷新恢复：`result_pending` 状态刷新后可恢复轮询并最终拿到结果。
