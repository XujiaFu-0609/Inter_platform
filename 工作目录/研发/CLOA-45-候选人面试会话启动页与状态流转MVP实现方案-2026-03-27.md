# CLOA-45 候选人面试会话启动页与状态流转 MVP 实现方案

## 1. 文档信息

- 对应任务：[CLOA-45](/CLOA/issues/CLOA-45)
- 上游依赖：[CLOA-42](/CLOA/issues/CLOA-42)、[CLOA-44](/CLOA/issues/CLOA-44)
- 父任务：[CLOA-43](/CLOA/issues/CLOA-43)

## 2. 目标与范围

### 2.1 目标

完成候选人进入面试前的统一启动页，并实现会话状态在前端的可见与可恢复流转，覆盖：

1. 待开始（pending）
2. 准备中（preparing）
3. 进行中（in_progress）
4. 异常中断（interrupted）
5. 已结束（completed）

### 2.2 本次 MVP 范围

1. 候选人通过面试链接进入启动页。
2. 前端发起会话创建或会话恢复。
3. 启动页展示当前状态、状态说明、下一步动作。
4. 前端通过轮询感知状态变化并驱动页面跳转。
5. 会话不存在/网络失败等异常具备重试与返回机制。
6. 启动页关键事件埋点覆盖进入、开始、失败重试。

### 2.3 不在本次范围

1. 面试中题目渲染与作答交互（由 [CLOA-46](/CLOA/issues/CLOA-46) 承接）。
2. 面试结束后的确认与结果提示（由 [CLOA-47](/CLOA/issues/CLOA-47) 承接）。
3. 音视频 SDK 深度接入与弱网自适应策略（后续迭代）。

## 3. 页面结构与交互

### 3.1 启动页信息区块

1. 顶部：岗位与场次基础信息（岗位名称、面试官、预估时长）。
2. 中部：状态卡片（状态标题、状态解释、当前步骤）。
3. 操作区：主按钮（开始/继续/重试）+ 次按钮（返回列表/联系支持）。
4. 底部：注意事项（摄像头/网络准备、隐私说明、异常处理提示）。

### 3.2 主要交互路径

1. 首次进入：自动创建会话，进入 pending 或 preparing。
2. 状态变为 in_progress：自动跳转面试进行页。
3. interrupted：停留在启动页并展示“恢复会话”与“重试检测”。
4. completed：提示已结束并跳转结束页。

## 4. 状态机设计

### 4.1 状态定义

- `pending`：会话已创建，等待资源就绪或面试官开始。
- `preparing`：系统正在准备题单/会话资源。
- `in_progress`：面试已开始。
- `interrupted`：会话因网络或服务异常中断，可恢复。
- `completed`：会话结束，不可继续。
- `not_found`：会话不存在或链接无效（前端异常态）。

### 4.2 状态流转规则

1. `pending -> preparing -> in_progress`
2. `pending/preparing/in_progress -> interrupted`
3. `interrupted -> pending/preparing/in_progress`（恢复成功）
4. `in_progress/interrupted -> completed`
5. 任意状态请求 404 时进入 `not_found`

### 4.3 前端路由动作

1. `pending/preparing/interrupted`：停留 `/session/:id/launch`
2. `in_progress`：跳转 `/session/:id/interview`
3. `completed`：跳转 `/session/:id/finish`
4. `not_found`：跳转 `/session/invalid`

## 5. 接口协作约定（对齐 [CLOA-44](/CLOA/issues/CLOA-44)）

### 5.1 期望接口

1. `POST /api/interview-sessions`
   - 入参：`candidateId`、`interviewPlanId`、`entryToken`
   - 出参：`sessionId`、`status`、`startedAt`
2. `GET /api/interview-sessions/{sessionId}`
   - 出参：`sessionId`、`status`、`updatedAt`、`errorCode?`
3. `POST /api/interview-sessions/{sessionId}/resume`
   - 出参：`status`

### 5.2 前端轮询策略

1. `pending/preparing`：每 3 秒轮询一次。
2. `interrupted`：每 5 秒轮询一次，同时允许手动重试。
3. 页面不可见（后台标签页）时放宽为每 10 秒。
4. 连续失败 3 次进入降级提示并展示“立即重试”。

## 6. 异常与可恢复策略

### 6.1 会话不存在

1. 展示“链接失效或会话不存在”说明。
2. 提供“返回面试列表”与“联系招聘支持”。
3. 埋点记录错误码与入口 token 哈希。

### 6.2 网络失败

1. 首次失败提示弱网并自动重试。
2. 连续 3 次失败弹出错误条并允许手动重试。
3. 重试成功后提示“连接已恢复”。

### 6.3 状态异常

当接口返回未知状态值时：

1. 前端降级为 `interrupted`。
2. 显示“状态同步异常，请重试”。
3. 上报告警埋点并附带原始状态值。

## 7. 埋点设计（MVP）

### 7.1 事件列表

1. `launch_page_view`
   - 触发：进入启动页
   - 字段：`sessionId`、`entrySource`、`networkType`
2. `launch_start_click`
   - 触发：点击开始或继续
   - 字段：`sessionId`、`currentStatus`
3. `launch_retry_click`
   - 触发：点击重试
   - 字段：`sessionId`、`retryReason`
4. `launch_status_changed`
   - 触发：轮询检测到状态变化
   - 字段：`fromStatus`、`toStatus`、`durationMs`
5. `launch_error_exposed`
   - 触发：用户可见错误提示
   - 字段：`errorType`、`errorCode`、`requestId`

### 7.2 埋点质量要求

1. 所有事件带 `sessionId`。
2. 关键错误事件带后端 `requestId`。
3. 事件发送失败不影响主流程。

## 8. 验收清单

1. 候选人可从有效入口创建并进入会话启动页。
2. 页面可正确展示 `pending/preparing/in_progress/completed/interrupted` 状态。
3. 状态变为 `in_progress` 后可自动跳转面试中页面。
4. 接口 404、网络中断、连续失败场景下均有可恢复提示与重试。
5. `launch_page_view`、`launch_start_click`、`launch_retry_click` 三类事件可在日志中查验。

## 9. 实施拆分与工时（2.5 人日）

1. 0.5 人日：页面框架与状态卡组件。
2. 0.75 人日：状态机与路由流转。
3. 0.5 人日：轮询与异常恢复。
4. 0.5 人日：埋点接入与联调。
5. 0.25 人日：验收走查与缺陷收敛。
