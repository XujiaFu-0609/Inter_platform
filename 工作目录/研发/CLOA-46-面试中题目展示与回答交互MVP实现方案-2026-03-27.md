## 2. 目标与范围

### 2.1 目标

完成候选人面试进行中的题目展示、回答输入、保存与提交反馈闭环，覆盖：

1. 题目顺序浏览与当前进度可见；
2. 回答草稿保存与提交反馈完整；
3. 页面刷新后恢复当前题目与已输入内容；
4. 关键交互埋点可用于联调与验收。

### 2.2 本次 MVP 范围

1. 在 `/session/:id/interview` 页面展示会话题目快照列表。
2. 支持单题回答输入（文本题、代码题统一以字符串内容承载）。
3. 支持手动保存与自动保存（防抖）两种草稿保存方式。
4. 支持题目切换时的未保存保护与保存提示。
5. 支持“提交本场回答”确认流程与成功/失败反馈。
6. 支持本地缓存恢复（当前题目索引、未落库草稿）。
7. 覆盖切题、保存失败、提交成功/失败三类关键埋点。

### 2.3 不在本次范围

1. 启动页状态流转（由 [CLOA-45](/CLOA/issues/CLOA-45) 承接）。
2. 结束页与结果提示（由 [CLOA-47](/CLOA/issues/CLOA-47) 承接）。
3. 代码执行样例运行与判题（后续版本由执行适配层承接）。
4. 协同作答或多端实时同步（后续迭代）。

## 3. 页面结构与交互

### 3.1 页面布局

1. 顶部信息栏：会话标题、剩余时间（若有）、整体进度（如 `2/5`）。
2. 左侧题目导航：按顺序展示题号、作答状态（未作答/已保存/保存失败）。
3. 中央题干区：展示题目标题、题干、约束、示例。
4. 右侧作答区：输入编辑器、字数统计、最近保存时间、错误提示。
5. 底部操作区：上一题、下一题、保存草稿、提交回答。

### 3.2 关键交互路径

1. 进入页面：先拉取题目列表与已保存回答，再定位当前题目。
2. 输入回答：内容变更后进入“未保存”状态，触发自动保存倒计时。
3. 切换题目：若存在未保存内容，先尝试保存；保存失败时阻止切换并提示。
4. 手动保存：点击保存后反馈“保存中/保存成功/保存失败”。
5. 提交回答：二次确认后提交，成功跳转结束页，失败停留当前页并提示可重试。

## 4. 前端状态模型

### 4.1 页面级状态

- `loading`：初始化加载题目与回答。
- `ready`：可正常作答。
- `saving`：当前题目保存中。
- `save_failed`：最近一次保存失败，可重试。
- `submitting`：全局提交中。
- `submit_failed`：提交失败，支持再次提交。

### 4.2 单题作答状态

- `pristine`：未修改且与服务端一致。
- `dirty`：已修改未保存。
- `saved`：已保存成功。
- `error`：保存失败。

### 4.3 状态流转规则

1. `loading -> ready`
2. `ready(dirty) -> saving -> saved`
3. `saving -> save_failed -> saving`（重试）
4. `ready -> submitting -> completed(路由跳转)`
5. `submitting -> submit_failed -> submitting`（重试）

## 5. 数据持久化与恢复策略

### 5.1 服务端持久化

1. 每题回答通过 `PUT /api/v1/sessions/{sessionId}/answers/{questionId}` 覆盖保存。
2. 页面初始化通过 `GET /api/v1/sessions/{sessionId}/answers` 恢复已保存内容。
3. 提交通过 `POST /api/v1/sessions/{sessionId}/submit` 触发会话进入 `reviewing`。

### 5.2 本地缓存策略

本地缓存键：`interview:{sessionId}:drafts`

缓存内容：

1. `currentQuestionId`
2. `draftByQuestionId`（仅保存未落库内容）
3. `updatedAt`

写入策略：

1. 输入变更后立即更新内存状态；
2. 每次自动保存触发前同步本地缓存；
3. 保存成功后清理对应题目的本地脏数据；
4. 提交成功后清理整个会话缓存。

### 5.3 恢复优先级

1. 以服务端已保存回答为主；
2. 若本地存在更新更晚的脏草稿，提示“检测到未同步草稿，可恢复”；
3. 用户确认后用本地草稿覆盖编辑区，并立即触发保存。

## 6. 接口协作约定（对齐 [CLOA-44](/CLOA/issues/CLOA-44)）

### 6.1 期望接口字段

1. `GET /api/v1/sessions/{sessionId}/questions`
   - 返回：`questions[]`（`questionId`、`orderNo`、`type`、`stem`、`constraints`）
2. `GET /api/v1/sessions/{sessionId}/answers`
   - 返回：`answers[]`（`questionId`、`answerContent`、`answerFormat`、`savedAt`、`answerVersion`）
3. `PUT /api/v1/sessions/{sessionId}/answers/{questionId}`
   - 请求：`answerContent`、`answerFormat`、`clientSavedAt`
   - 返回：`answerVersion`、`savedAt`
4. `POST /api/v1/sessions/{sessionId}/submit`
   - 返回：`status=reviewing`、`submittedAt`

### 6.2 前端保存策略

1. 自动保存：输入停止 1200ms 后触发。
2. 切题前保存：存在 `dirty` 数据时先执行保存。
3. 并发保护：同一题保存未完成前，新的保存请求仅保留最后一次。
4. 超时处理：单次保存超过 8 秒视为失败，进入 `save_failed`。

## 7. 异常处理策略

### 7.1 题目加载失败

1. 首次失败展示“题目加载失败，请重试”。
2. 提供“重新加载”按钮。
3. 连续 3 次失败时展示支持入口并保留 requestId。

### 7.2 保存失败

1. 在当前题显示错误标签与最近失败时间。
2. 不丢失编辑区内容，保留 `dirty` 状态。
3. 允许“再次保存”与“切换前重试保存”。

### 7.3 提交失败

1. 弹出错误提示并展示失败原因（若后端返回 errorCode）。
2. 保留当前答案与题目位置，允许立即重试提交。
3. 若返回 `SESSION_STATE_CONFLICT`，先刷新会话状态并提示可能已提交。

### 7.4 页面刷新与断线恢复

1. 刷新后自动恢复到上次题目位置。
2. 若本地存在未同步草稿，显示恢复提示条。
3. 网络恢复后优先触发当前题保存，再允许提交。

## 8. 埋点设计（MVP）

### 8.1 事件列表

1. `interview_question_switched`
   - 触发：切换题目成功
   - 字段：`sessionId`、`fromQuestionId`、`toQuestionId`、`triggerType`
2. `interview_answer_save_attempt`
   - 触发：发起保存（自动/手动）
   - 字段：`sessionId`、`questionId`、`saveMode`、`contentLength`
3. `interview_answer_save_failed`
   - 触发：保存失败
   - 字段：`sessionId`、`questionId`、`errorCode`、`requestId`
4. `interview_answer_submit_attempt`
   - 触发：点击确认提交
   - 字段：`sessionId`、`answeredCount`、`totalCount`
5. `interview_answer_submit_result`
   - 触发：提交返回成功或失败
   - 字段：`sessionId`、`result`、`errorCode?`、`requestId?`

### 8.2 埋点质量要求

1. 所有事件必须带 `sessionId` 与页面时间戳。
2. 失败事件必须带 `errorCode`（如有）与 `requestId`（如有）。
3. 埋点发送失败不得阻塞保存与提交主链路。

## 9. 验收清单

1. 页面可按顺序展示题目、题干、输入区与进度信息。
2. 回答保存与提交具备明确成功/失败反馈。
3. 页面刷新后可恢复当前题目与已输入内容（服务端恢复 + 本地兜底）。
4. 切题、提交、保存失败均有可验证埋点日志。
5. 至少通过 4 组联调用例：
   - 正常切题与保存
   - 保存失败后重试成功
   - 提交成功并跳转结束页
   - 提交失败停留并重试

## 10. 实施拆分与工时（3 人日）

1. 0.75 人日：页面骨架、题目导航、进度展示。
2. 0.75 人日：作答状态机与自动保存机制。
3. 0.5 人日：本地缓存恢复与刷新兜底。
4. 0.5 人日：提交确认流与异常处理。
5. 0.5 人日：埋点接入、联调走查与缺陷收敛。
