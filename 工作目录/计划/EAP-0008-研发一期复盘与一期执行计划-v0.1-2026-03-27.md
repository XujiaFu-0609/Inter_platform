## [任务理解]

### 1. 目标

本期目标不是继续堆演示功能，而是把 AI Infra 面试平台从“Go 单体演示版”收敛到“Go 后端 + Kubernetes Operator 控制面 + Spark/Flink 数据面 + Volcano 调度”的一期可落地方案，并完成明确派工、里程碑和验收口径。

### 2. 范围

1. 给出 AI Infra 面试平台一期总体技术架构。
2. 复核当前后端实现是否符合 Go + Operator 方向，并给出重构路径。
3. 明确 Spark/Flink 与 Volcano 融合方案，以及可视化与可观测交付边界。
4. 将工作拆分到 Big Data Architect、Backend Engineer、Frontend Engineer、Cloud Native Architect、QA Lead。

### 3. 强约束

1. 后端正式实现必须使用 Go。
2. 控制面优先采用 Kubernetes Operator 模式，使用 CRD/Controller 管理平台资源。
3. 中间件优先 Redis、PostgreSQL、Kafka。
4. 流批引擎必须采用 Spark/Flink，并与 Volcano 调度整合。
5. 禁止引入 YARN 作为资源调度底座。

### 4. 非目标

1. 本周内不追求一次性交付完整生产级多租户数据平台。
2. 不将高频业务状态机直接下沉到 Kubernetes Controller 中。
3. 不继续扩展 Node.js 作为正式后端实现。

### 5. 关键假设

1. 当前仓库中的 Go 服务仅用于最小主链路演示，允许按模块化单体 + Operator 控制面方式继续演进。
2. 一期环境默认基于单 Kubernetes 集群推进，Spark/Flink 以 K8s 原生模式运行并接入 Volcano。
3. 一期先完成架构、控制面骨架、运行基线和验收清单，不强求在 2026-04-03 前完成全部生产化细节。

## [总体架构]

### 1. 应用层

1. 候选人端前端：面试启动、答题、提交、结果态展示。
2. 管理端/平台端：作业模板、运行状态、任务看板、告警总览。
3. Go Interview API：对外提供 canonical API，负责会话编排、题目读取、答案记录、评测触发、结果查询。
4. Admin API：对外暴露平台配置、作业提交流程、作业状态聚合接口。

### 2. 数据层

1. PostgreSQL：保存面试会话、题库元数据、评测结果索引、平台配置。
2. Redis：保存会话缓存、幂等键、热点查询缓存、短时状态加速。
3. Kafka：承载答题事件、评测事件、运行事件、审计事件。
4. 对象存储/湖仓层（如 MinIO + Iceberg/Hive Metastore）：承载 Spark/Flink 的离线与准实时数据落地。
5. Grafana + Prometheus + Loki：承载指标、日志、链路与运行可视化。

### 3. 控制面

控制面全部使用 Go 实现，并以 Operator 模式运行在 Kubernetes 中：

1. `InterviewPlatform` CRD：管理 API、Web、Config、Secret、依赖连接与版本收敛。
2. `StreamPipeline` CRD：定义 Flink 作业模板、Kafka Topic、Checkpoint、SLA、队列优先级。
3. `BatchPipeline` CRD：定义 Spark 作业模板、输入输出数据集、重试策略、队列与资源档位。
4. `ObservabilityStack` CRD：收敛 Grafana 数据源、Dashboard、告警规则和日志采集配置。
5. Controller 仅管理平台资源编排、版本与策略，不承载面试业务状态机。

### 4. 运行面

1. Kubernetes 作为统一运行底座。
2. Volcano 作为统一调度器，负责 Queue、PodGroup、优先级、抢占与容量治理。
3. Spark on Kubernetes 承担离线与批处理任务。
4. Flink on Kubernetes 承担实时流处理、实时聚合和指标加工。
5. 所有 Spark/Flink 任务通过 `schedulerName: volcano` 与队列治理对接。

### 5. 当前实现复盘结论

当前 `backend/cmd/interview-api/main.go` 与 `backend/internal/httpapi/server.go` 已满足 Go 主链路演示，但仍存在以下缺口：

1. 运行形态仍是单 Go HTTP 进程，不是 Operator 控制面。
2. 会话数据仍以内存态为主，尚未接入 PostgreSQL / Redis。
3. 事件处理仍是进程内队列，尚未切换到 Kafka。
4. 尚未定义任何 CRD/Controller，也未与 Volcano、Spark、Flink 打通。
5. 目前仅能证明“主链路可跑”，不能证明“云原生平台可交付”。

## [关键路径与并行计划]

### 1. 关键路径

1. 控制面边界冻结 → 后端重构路径冻结 → Spark/Flink/Volcano 场景映射 → 运行与观测基线冻结 → QA 验收口径冻结。
2. 如果 `InterviewPlatform` / `BatchPipeline` / `StreamPipeline` CRD 边界不先冻结，后续后端、前端、QA 都会出现返工。
3. 因此 Cloud Native Architect 与 Backend Engineer 是首要关键路径，Big Data Architect 紧随其后。

### 2. 并行计划

1. Backend Engineer 与 Cloud Native Architect 在 2026-03-28 前并行完成“服务边界”与“控制面边界”。
2. Big Data Architect 在同一时间窗并行完成 Spark/Flink/Volcano 任务模型与队列策略。
3. Frontend Engineer 在 2026-03-29 起基于冻结后的 API/可视化字段并行出页面草图。
4. QA Lead 从 2026-03-28 起提前介入，不等待研发全部完成后再补验收。

### 3. 里程碑

1. D1（2026-03-28）：完成总体架构冻结、CRD 草案、后端差距分析、批流场景映射。
2. D3（2026-03-30）：完成后端重构设计、Volcano 队列治理方案、可视化草图、运行与观测基线。
3. D7（2026-04-03）：完成一期联调包、验收清单、风险收口结论和下一阶段实施建议。

## [风险与缓解]

### Top 5 风险

1. **后端继续按单体演示代码扩张**  
   缓解：先冻结服务分层与 Operator 边界，新增能力必须按模块化单体 + 控制面设计接入。
2. **Spark/Flink 与 Volcano 集成口径不统一**  
   缓解：由 Big Data Architect 输出统一任务模板与队列治理规则，禁止各自为战。
3. **把业务状态机错误地下沉到 CRD/Controller**  
   缓解：Controller 仅管理资源与配置；业务态继续由 Go 服务持有。
4. **Grafana 可视化与业务前端重复建设**  
   缓解：前端承载业务态看板，平台监控与底层运行指标优先落 Grafana。
5. **QA 后置介入导致 D7 暴露系统性返工**  
   缓解：QA 从 D1 起绑定关键评审，采用证据模板前置验收。
