## [测试策略]

1. 以“早介入、强门禁、可追溯”为原则，按 D1/D3/D7 分阶段验收，不等待开发收尾后集中测试。
2. 覆盖六类测试：功能测试、接口测试、集成测试、性能测试、稳定性测试、回归测试。
3. 四条主线统一口径：
   - Go API：接口语义、幂等、错误码、追踪字段；
   - Operator/CRD：状态收敛、幂等 reconcile、事件与回滚；
   - Spark/Flink/Volcano：任务提交流程、队列治理、异常恢复；
   - Grafana 可观测：指标完整性、告警可达、日志可检索。
4. 风险优先级：
   - P0（阻塞上线）：主链路不可用、数据错误、不可回滚；
   - P1（高风险）：关键场景降级、SLA 不达标、监控盲区；
   - P2（中风险）：非关键功能缺陷或可替代方案存在；
   - P3（低风险）：文档、体验与轻微展示问题。
5. 发布策略：实行 QA + CTO 双门禁，任一门禁不通过则结论为 No-Go。

## [自动化与环境计划]

1. 自动化分层
   - PR 级：API 冒烟 + 契约检查 + 核心单测（10 分钟内）；
   - 日构建：核心回归（接口、集成、可观测）；
   - 周构建：稳定性（长稳 4h）+ 性能观察（基线对比）。
2. 环境分层
   - Env-A（开发联调）：本地/Kind，快速冒烟与契约校验；
   - Env-B（集成验证）：K8s 集群 + Operator + 消息总线；
   - Env-C（准发布）：仿真生产配置，用于 D7 最终回归与发布评估。
3. 数据计划
   - 建立固定样例数据集（正常、边界、异常三类）；
   - 每次执行记录 `requestId` 作为证据主键；
   - 缺陷复现数据沉淀到 `evidence/` 并附执行脚本。
4. 协同机制（后端/前端/云原生）
   - 每日 10:00 站会同步阻塞项；
   - D1、D3、D7 当天 18:00 里程碑验收会；
   - 统一使用“用例ID-缺陷ID-证据路径”三元关联。

## [质量门禁与验收标准]

1. 缺陷分级与处理时限
   - P0：2 小时内响应，24 小时内给出修复或回滚方案；
   - P1：4 小时内响应，48 小时内修复或给出可接受豁免；
   - P2：纳入迭代修复，不阻塞 D 阶段验收；
   - P3：跟随版本优化。
2. 发布门禁（D7）
   - 功能/接口/集成主链路通过率 100%；
   - 性能与稳定性无 P0/P1 未决项；
   - Grafana 核心告警链路演练通过；
   - 回滚脚本与值班联系人明确；
   - QA 与 CTO 均给出 Go 结论。
3. 阻塞升级路径
   - 触发条件：出现 P0 或连续 2 次里程碑未达标；
   - 升级路径：QA Lead -> CTO -> CEO/Board；
   - 升级动作：标记 `blocked`、创建 War Room、明确 owner 与时限。
4. Go/No-Go 判定规则
   - Go：满足全部门禁；
   - Conditional Go：无 P0，且仅存在有兜底措施的少量 P1（仅 D1/D3 可用）；
   - No-Go：存在任一 P0，或关键回归/回滚能力未通过。
