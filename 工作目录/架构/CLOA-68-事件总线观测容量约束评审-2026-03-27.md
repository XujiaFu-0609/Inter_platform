## 2. 评审范围与结论

本评审聚焦四个问题：

1. NATS 事件广播与审计链路是否满足当前阶段；
2. `requestId`、`traceId`、指标采集、容量看板如何打通；
3. Go 后端切换后对日志、指标、队列、成本带来的新增约束；
4. 下一阶段可执行动作与 CTO 决策项。

总体结论：

1. **当前阶段可继续采用 NATS 作为事件总线**，满足 MVP 的事件广播与低延迟解耦诉求；
2. 现有文档已定义 `requestId`/`traceId` 目标，但**缺少“事件级追踪主键 + 指标标签规范”**，会导致排障链路断裂；
3. Go + Kubernetes + Operator 路线可落地，但需要先补齐**事件 schema、观测采样策略、容量配额闭环与成本归集口径**，否则会出现“链路可跑但不可治理”风险。

## 3. 对 NATS / 事件广播 / 审计链路的评审

### 3.1 满足当前阶段的点

1. MVP 事件类型规模有限（题目下发、提交、面评、回流、审计），NATS 可满足轻量发布订阅与快速接入；
2. 部署方案已将 `nats` 纳入标准拓扑，便于与 `api` 同集群治理；
3. 对 Go 服务侧，NATS 客户端成熟，接入成本低，便于快速形成事件驱动闭环。

### 3.2 当前缺口

1. 文档虽规定 NATS subject 命名规范，但未冻结**事件 payload 最小字段集**；
2. 未明确审计事件是否走 JetStream 持久化，存在“只广播不留痕”风险；
3. 事件重放、幂等消费、死信回收尚无统一策略；
4. 未定义事件版本演进策略（`schemaVersion`、兼容窗口、弃用流程）。

### 3.3 评审判断

NATS 方案短期可用，但应明确边界：**MVP 先保证“可追踪 + 可回放 + 可审计”，再讨论高吞吐扩展**。若后续出现离线分析吞吐显著增长，可增量引入 Kafka 作为分析通道，不替换在线主总线。

## 4. `requestId` / `traceId` / 指标 / 容量看板衔接评审

### 4.1 推荐统一链路键模型

1. **请求层**：`requestId`（HTTP 入站生成）；
2. **链路层**：`traceId` / `spanId`（OTel 标准）；
3. **事件层**：`eventId`、`correlationId=requestId`、`traceId`；
4. **容量层**：指标标签最少包含 `service`、`queue`、`eventType`、`result`、`costCenter`。

### 4.2 最小观测闭环（必须补齐）

1. 日志：结构化 JSON，统一字段 `timestamp/requestId/traceId/eventId/sessionId`；
2. 指标：新增事件投递成功率、消费滞后、死信量、重试次数、队列深度；
3. 链路：HTTP 入站 -> 业务处理 -> NATS publish/consume 全链路 Trace；
4. 看板：L0/L1 保留既有容量视角，新增“事件管道健康”分面（发布失败率、消费延迟 P95、DLQ 增速）。

### 4.3 风险点

1. 仅保留 `requestId` 不足以覆盖异步链路问题定位；
2. 仅有业务 SLA 看板而无事件管道看板，会导致“接口正常但异步堆积”不可见；
3. 未做统一标签治理将导致成本归集与容量预测不可比对。

## 5. Go 切换后的新增约束（日志/指标/队列/成本）

### 5.1 日志约束

1. Go 服务统一采用结构化日志库，禁止纯文本日志作为主输出；
2. 错误日志必须包含 `errorCode` 与 `requestId`，异步处理还需包含 `eventId`；
3. 日志采样要区分成功/失败链路，失败链路禁止降采样。

### 5.2 指标约束

1. 统一 OTel + Prometheus 指标暴露规范；
2. 新增事件总线 SLI：发布成功率、消费成功率、P95 处理时延、DLQ 速率；
3. 关键指标必须绑定队列与工作负载标签，支持与 EAP-0005 容量看板联动。

### 5.3 队列与容量约束

1. NATS topic/stream 需按业务域分层，避免单一主题过热；
2. 消费组并发上限要与 Kubernetes HPA / 资源配额联动，避免“扩容放大重试风暴”；
3. 回流与审计消费优先级需低于主流程，防止抢占核心资源。

### 5.4 成本约束

1. 事件风暴场景会推高网络与存储成本，需设定事件保留与压缩策略；
2. 日志全量采集在高并发下成本增长显著，应按业务等级定义采样与保留期；
3. 需要把事件处理成本纳入队列级成本看板（与 GPU/CPU 成本同口径汇总）。

## 6. 可执行建议（含 CTO 决策项）

1. **冻结事件最小 schema（本周）**  
   字段至少包含：`eventId`、`eventType`、`schemaVersion`、`occurredAt`、`requestId`、`traceId`、`sessionId`、`payload`。  
   - 责任：Backend Engineer + Big Data Architect  
   - **CTO 决策**：是否将 JetStream 持久化设为强制要求。

2. **补齐事件观测看板（T+7 天）**  
   在现有 L0/L1 看板增加事件通道健康面板，并定义 P2/P1/P0 阈值。  
   - 责任：Senior Cloud Native Operations Engineer  
   - 协作：QA Lead 设计异常注入回归场景

3. **定义异步幂等与死信治理标准（T+7 天）**  
   统一消费幂等键、重试上限、死信回放流程，形成 runbook。  
   - 责任：Backend Engineer + 运维  
   - **CTO 决策**：是否允许跨域共享死信主题，还是按域隔离。

4. **建立容量-成本联动报表（T+14 天）**  
   把事件处理量、消费滞后、资源利用率、成本偏差并表输出。  
   - 责任：Big Data Architect  
   - 协作：运维提供成本与监控数据接入

5. **明确 NATS 与 Kafka 分工边界（本周架构确认）**  
   NATS 负责在线业务事件；Kafka（如引入）仅承接离线分析/长周期回放。  
   - **CTO 决策**：是否在当前阶段即预留 Kafka 接入位。
