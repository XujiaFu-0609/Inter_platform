# Cloud_Native 架构基线文档

日期：2026-03-26  
责任人：Cloud Native Architect  
状态：基线版本 v1.0

## 1. 目标与边界

### 1.1 目标

本基线用于统一 Cloud_Native 项目后续架构输出的最小共识，确保研发与运维围绕同一套约束推进实施。

### 1.2 适用边界

1. 本文适用于当前仓库及 `工作目录/` 下所有正式产出。
2. 本文聚焦平台与工程架构基线，不替代业务详细设计文档。
3. 本文作为后续 ADR、实施计划、运维手册的输入前提。

### 1.3 不在本文范围

1. 具体业务域模型与接口细节设计。
2. 特定云厂商资源选型的最终采购结论。
3. 详细项目排期与人力预算拆解。

## 2. 工作目录结构约束

## 3. 关键基础设施与平台假设

以下为当前阶段默认假设，后续如决策变更需通过 ADR 记录并回溯更新本文。

1. **运行平台假设**：核心服务将以容器化方式运行，目标平台为 Kubernetes 体系（开发、测试、生产环境分层）。
2. **交付链路假设**：代码通过 Git 管理，后续建立最小 CI/CD 流水线（构建、测试、镜像发布、部署）。
3. **配置与密钥假设**：应用配置与敏感信息分离管理，密钥不入库，采用集中密钥管理机制。
4. **可观测性假设**：默认建设日志、指标、链路追踪三类可观测能力，并设立基础告警规则。
5. **环境一致性假设**：开发到生产尽量保持镜像、部署模板、配置结构一致，仅通过参数区分环境。

## 4. 已知风险与待决策项

| 类别 | 当前风险/待决策 | 影响 | 需要协同角色 |
| --- | --- | --- | --- |
| 平台选型 | Kubernetes 发行版与托管方式尚未确认（自建/托管） | 影响部署模型、运维成本与团队技能要求 | 架构、运维、管理层 |
| 交付治理 | CI/CD 工具链与发布门禁标准未统一 | 影响发布稳定性与变更可追溯性 | 研发、运维 |
| 可观测性 | SLI/SLO 与告警分级未定义 | 影响故障发现速度与值班质量 | 运维、业务负责人 |
| 安全合规 | 密钥管理与镜像安全扫描流程未落地 | 影响供应链安全与审计合规 | 安全、运维、研发 |
| 目录治理 | 架构产出模板尚未标准化 | 影响文档质量一致性与复用效率 | 架构、研发 |

## 5. 优先推进的架构工作项（1-3）

1. **P1：完成平台目标架构与环境分层 ADR**
   - 产出：目标架构图、环境分层说明、关键组件职责与边界。
   - 价值：为研发部署、运维接管、容量规划提供统一蓝图。
2. **P2：建立交付基线（IaC + CI/CD 最小闭环）**
   - 产出：基础流水线模板、部署模板、发布门禁清单。
   - 价值：将“可构建、可部署、可回滚”作为项目默认能力。
3. **P3：定义可观测性与运维基线**
   - 产出：日志规范、核心指标清单、告警分级与值班响应流程。
   - 价值：提升运行稳定性，降低故障定位与恢复时间。

## 6. 对研发与运维执行的支持建议

1. 以本文为上游约束，研发侧优先补齐模块边界和部署单元设计，避免与平台模型冲突。
2. 运维侧基于“环境分层 + 发布门禁 + 可观测性”三条主线同步建设，避免后期补救式治理。
3. 新增架构决策统一沉淀到 `工作目录/架构/`，并在对应研发/运维文档中引用，确保执行与架构一致。
