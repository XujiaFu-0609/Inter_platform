## 2. 参考架构（Kubernetes + Volcano）

### 2.1 分层架构

1. 数据接入层：CDC、消息总线、对象存储事件。
2. 数据存储与元数据层：Lakehouse（Parquet/ORC）、Hive Metastore、统一 Catalog。
3. 计算引擎层：
   - Spark：离线 ETL、批处理特征加工。
   - Flink：实时计算、流批融合链路。
   - Hive：SQL 交互分析、离线数仓治理任务。
4. 调度治理层（Volcano）：
   - Queue：业务域与租户资源边界。
   - PodGroup：关键任务整组调度保证。
   - PriorityClass + preempt/reclaim：SLA 分级与资源保护。
5. 平台治理与可观测层：指标、日志、事件、成本归集、变更审计。

### 2.2 关键接口与职责

1. 引擎团队负责 Runtime 参数、作业模板、性能优化。
2. 平台团队负责 Volcano 队列策略、准入门禁、容量基线。
3. 数据治理团队负责表规范、数据质量规则、权限策略。

## 3. 演进路线图（12 个月）

### 阶段 A（0-2 个月）：底座标准化

1. 统一 Spark/Flink/Hive 作业模板与标签规范。
2. 固化 `prod-realtime`、`prod-batch`、`ai-train`、`dev-sandbox` 四队列基线。
3. 建成调度与容量核心看板（排队时长、成功率、资源利用率）。

### 阶段 B（2-6 个月）：生产链路迁移

1. Spark 离线主链路迁移至 Volcano，完成资源隔离。
2. Flink 实时链路引入 SLA 分级告警与抢占保护。
3. Hive 批量 SQL 任务纳入统一队列门禁与成本归档。

### 阶段 C（6-9 个月）：优化与自动化

1. 引入按业务时段的弹性配额与回收策略。
2. 建立容量预测模型，联动季度预算与采购计划。
3. 将变更准入、回滚演练、SLO 验收标准自动化。

### 阶段 D（9-12 个月）：智能治理

1. 基于历史负载做智能排队建议与风险预警。
2. 引入跨引擎统一成本核算与单位任务成本优化。
3. 建立大数据域架构评审委员会与年度技术雷达。

## 4. 关键技术取舍

| 主题 | 方案选择 | 选择原因 | 暂不采用原因 |
|---|---|---|---|
| 统一调度 | Volcano 统一 Spark/Flink/Hive 调度 | 与 K8s 原生兼容，支持队列治理与抢占回收 | 多套调度体系并行会增加治理复杂度 |
| 存储格式 | Parquet/ORC 双格式并存 | 兼容既有数仓与流批查询场景 | 单一格式难兼顾查询与写入效率 |
| 元数据 | Hive Metastore + Catalog 统一入口 | 平滑承接历史资产，降低迁移成本 | 自建新 Catalog 一次性替换风险高 |
| 资源治理 | 四队列分层 + 分级 SLA | 与 [CLOA-24](/CLOA/issues/CLOA-24) 既有基线一致 | 纯按团队划分队列会弱化业务优先级 |

## 5. 与运维、QA 的协作边界

1. 运维负责：集群容量、节点健康、调度组件高可用、基础告警路由。
2. 架构/平台负责：队列策略、准入规则、回滚策略、变更评审。
3. QA 负责：策略变更回归矩阵、SLA 验证、故障演练验收。
4. 三方共担：季度容量复盘、P0/P1 事故 RCA 与治理项闭环。

## 6. 可拆分实施项（Backlog）

1. `BD-ARCH-01`：统一 Spark/Flink/Hive 作业模板与发布门禁。
2. `BD-ARCH-02`：建立四队列资源仪表盘与月度容量报表。
3. `BD-ARCH-03`：完成 Flink 关键链路 SLA 分级与告警联动。
4. `BD-ARCH-04`：沉淀 Hive 任务资源画像与夜间窗口调度策略。
5. `BD-ARCH-05`：输出季度架构评审模板与技术债跟踪清单。

## 7. 未决风险

1. 业务高峰预测误差可能导致 `prod-realtime` 保底资源不足。
2. 历史 Hive 任务资源申报不准确会影响容量模型精度。
3. 多引擎版本升级窗口重叠时，调度策略回滚压力上升。
