# EAP-0005 队列治理与容量模型

日期：2026-03-27  
责任人：Big Data Architect  
关联任务：[CLOA-24](/CLOA/issues/CLOA-24)、[CLOA-36](/CLOA/issues/CLOA-36)  
对齐文档：[EAP-0005-Volcano与大数据融合落地文档](./EAP-0005-Volcano与大数据融合落地文档.md)

## 1. 目标说明

在 [CLOA-24](/CLOA/issues/CLOA-24) 既有队列基线基础上，固化四队列容量阈值、抢占/回收规则、变更准入与季度复盘口径，作为生产执行标准。

## 2. 四队列容量基线（与 CLOA-24 一致）

| 队列 | 目标负载 | guarantee（保底） | capacity（上限） | 说明 |
|---|---|---|---|---|
| `prod-realtime` | 实时核心链路 | CPU 35%，内存 35%，GPU 20% | CPU 60%，内存 60%，GPU 35% | 最高优先级，保障线上时延 |
| `prod-batch` | 离线 ETL/报表 | CPU 25%，内存 25%，GPU 10% | CPU 50%，内存 50%，GPU 20% | 夜间可借用空闲资源 |
| `ai-train` | 训练与大模型批任务 | CPU 20%，内存 20%，GPU 40% | CPU 45%，内存 45%，GPU 70% | 区分可中断/不可中断任务 |
| `dev-sandbox` | 开发测试 | CPU 5%，内存 5%，GPU 0% | CPU 20%，内存 20%，GPU 10% | 始终可被抢占回收 |

> 说明：本版参数未偏离 [CLOA-24](/CLOA/issues/CLOA-24) 基线；本轮不做参数修订，仅补齐治理与执行口径。

## 3. 抢占与回收规则

### 3.1 抢占优先级矩阵

1. `prod-realtime` 可抢占 `prod-batch` / `ai-train` / `dev-sandbox`。
2. `prod-batch` 仅可抢占 `dev-sandbox`。
3. `ai-train` 仅可回收可中断训练任务，不可抢占 `prod-realtime`。
4. `dev-sandbox` 不具备抢占权。

### 3.2 回收策略

1. 优先回收 `dev-sandbox` 与 `ai-train` 可中断作业。
2. 夜间窗口（22:00-08:00）允许 `prod-batch` 临时提升至上限。
3. 工作时段（08:00-22:00）若生产队列资源饥饿，立即触发回收。

## 4. 变更准入流程

1. 提交容量变更单：包含业务背景、峰值预测、回滚阈值、影响评估。
2. 架构评审：Big Data Architect + Cloud Native Architect 双签。
3. 运维预检：节点余量、配额冲突、监控项完整性检查。
4. 灰度发布：先 `dev-sandbox`，再 `prod-batch`，最后 `prod-realtime`。
5. 观察窗口：至少 1 个业务高峰周期，未命中阈值才可正式生效。

## 5. 季度复盘口径

1. 容量利用率：按队列统计 CPU/内存/GPU 平均与 P95 使用率。
2. SLA 达成率：按队列统计关键任务成功率、排队时长、超时率。
3. 成本偏差：预算与实际偏差控制在 ±10%。
4. 策略有效性：抢占命中率、误伤率、回滚次数。
5. 输出产物：季度容量报告、优化清单、下一季度参数建议。

## 6. 与运维、QA 的协作边界

1. 运维：执行容量变更、维护监控阈值、保障调度组件稳定。
2. 架构：制定参数基线、审批变更、解释偏差与风险。
3. QA：验证抢占/回收场景、回归关键链路、确认回滚可用性。
4. 联合机制：每季度召开一次容量治理评审会并形成行动项。

## 7. 可拆分实施项（Backlog）

1. `BD-QUEUE-01`：实现队列级容量看板与异常趋势自动告警。
2. `BD-QUEUE-02`：上线容量变更单模板与自动校验脚本。
3. `BD-QUEUE-03`：补齐 `ai-train` 可中断任务标签治理。
4. `BD-QUEUE-04`：建立抢占误伤自动识别与回滚触发器。
5. `BD-QUEUE-05`：形成季度容量复盘自动报表流水线。

## 8. 未决风险

1. 训练任务标签缺失将导致回收策略误判。
2. 峰值流量突发可能造成 `prod-batch` 夜间借用策略失衡。
3. GPU 供给波动会放大 `ai-train` 与实时推理的资源冲突。
