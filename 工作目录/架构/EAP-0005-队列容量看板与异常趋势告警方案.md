## 1. 目标与约束

- 目标：将四队列（`prod-realtime`、`prod-batch`、`ai-train`、`dev-sandbox`）容量治理基线落地为可执行看板与告警机制。
- 约束：队列容量、SLA 与阈值沿用 [CLOA-24](/CLOA/issues/CLOA-24) / [CLOA-36](/CLOA/issues/CLOA-36) 已确认参数，本方案不修改基线参数。
- 产出边界：覆盖指标体系、分层看板、趋势规则、告警升级链路、跨团队协作、rollout 计划。

## 2. 四队列容量与SLA基线（继承既有结论）

| 队列 | guarantee（保底） | capacity（上限） | SLA 目标（排队时长） |
|---|---|---|---|
| `prod-realtime` | CPU 35%，内存 35%，GPU 20% | CPU 60%，内存 60%，GPU 35% | P95 <= 30s |
| `prod-batch` | CPU 25%，内存 25%，GPU 10% | CPU 50%，内存 50%，GPU 20% | P95 <= 3min |
| `ai-train` | CPU 20%，内存 20%，GPU 40% | CPU 45%，内存 45%，GPU 70% | P95 <= 10min |
| `dev-sandbox` | CPU 5%，内存 5%，GPU 0% | CPU 20%，内存 20%，GPU 10% | P95 <= 30min |

## 3. 指标字典（Metric Dictionary）

| 指标编码 | 指标名称 | 定义/口径 | 维度 | 数据来源 | 刷新频率 | 主看板 |
|---|---|---|---|---|---|---|
| `Q_CAP_UTIL` | 容量利用率 | `used / capacity`（CPU/内存/GPU 分别统计） | queue, resource | Prometheus + kube-state-metrics | 1 分钟 | L1 队列容量 |
| `Q_GUARANTEE_GAP` | 保底偏离率 | `(guarantee - allocatable_to_queue) / guarantee` | queue, resource | Volcano scheduler metrics | 1 分钟 | L1 队列容量 |
| `Q_WAIT_P95` | 排队时长 P95 | 5 分钟滚动窗口 P95 | queue, sla_level | Volcano + 作业事件流 | 1 分钟 | L1/L2 |
| `Q_JOB_SUCCESS_15M` | 作业成功率（15m） | 15 分钟窗口 `success / total` | queue, engine | Spark/Flink/Hive 任务状态聚合 | 1 分钟 | L2 作业质量 |
| `Q_PREEMPT_COLLATERAL` | 抢占误伤率 | 抢占后关键作业失败占比（15m） | queue, task_class | 调度事件 + 作业状态 | 1 分钟 | L2 抢占健康 |
| `Q_GPU_COST_DEV` | GPU 成本偏差 | `(actual_cost - budget_cost) / budget_cost`（日） | queue | 成本账单 + 预算台账 | 15 分钟 | L1 经营视角 |
| `Q_PENDING_DEPTH` | 排队深度 | Pending 作业数量与等待资源总量 | queue, priority | Volcano 队列快照 | 1 分钟 | L1/L2 |
| `Q_QUOTA_BORROW` | 配额借用率 | 夜间借用资源占目标队列 capacity 比例 | queue | 配额控制器日志 | 5 分钟 | L2 借用治理 |

## 4. 分层看板设计

### 4.1 L0 总览看板（管理层/值班总览）

- 面板：四队列健康灯（绿/黄/红）、今日告警分级计数（P2/P1/P0）、GPU 成本偏差、当前升级事件。
- 刷新：1 分钟（成本面板 15 分钟）。
- 使用者：值班运维、研发负责人、Big Data Architect。

### 4.2 L1 队列容量看板（队列治理主视图）

- 面板：`Q_CAP_UTIL`（CPU/内存/GPU）、`Q_GUARANTEE_GAP`、`Q_WAIT_P95`、`Q_PENDING_DEPTH`、`Q_GPU_COST_DEV`。
- 视角：按队列对比 + 24h 趋势 + 峰谷分段（08:00-22:00、22:00-08:00）。
- 刷新：1 分钟。
- 阈值呈现：面板内置 P2/P1/P0 阈值线，超过阈值自动打点。

### 4.3 L2 异常诊断看板（值班/排障）

- 面板：`Q_PREEMPT_COLLATERAL`、抢占事件时间线、失败作业 TopN、借用/回收动作时间线、近期阈值变更记录。
- 过滤：队列、引擎（Spark/Flink/Hive）、任务等级（S0-S3）。
- 刷新：1 分钟（阈值变更记录 5 分钟）。

### 4.4 L3 复盘看板（月度治理）

- 面板：月度容量利用率箱线图、告警收敛时长分布、误伤率趋势、成本偏差趋势、RCA 闭环率。
- 刷新：每日 1 次（T+1），供月度复盘会使用。

## 5. 异常趋势判断规则

1. **短时突变规则（Spike）**：5 分钟内斜率超过 3 倍历史同时间窗均值，标记“突发异常”。
2. **持续劣化规则（Degrade）**：连续 3 个窗口（每窗口 5 分钟）同向劣化且未回落至 P2 阈值以下。
3. **周期漂移规则（Drift）**：近 7 天同时间点中位数较基线偏移超 15%，标记“趋势漂移”。
4. **关联触发规则（Correlation）**：当 `Q_WAIT_P95` 与 `Q_PENDING_DEPTH` 同时超阈且 `Q_GUARANTEE_GAP` 恶化，升级一级处理优先级。

## 6. 告警分级设计（四类必选告警）

### 6.1 容量逼近告警（Capacity Near Limit）

- 指标：`Q_CAP_UTIL`、`Q_GUARANTEE_GAP`。
- 触发：
  - P2：任一生产队列 `Q_CAP_UTIL` > 85% 持续 10 分钟，或保底偏离 > 5%。
  - P1：任一生产队列 `Q_CAP_UTIL` > 92% 持续 10 分钟，或保底偏离 > 10%。
  - P0：`prod-realtime` `Q_CAP_UTIL` > 97% 持续 10 分钟，或保底偏离 > 15%。
- 收敛：连续 15 分钟低于对应阈值且 Pending 深度回到基线区间。
- 升级路径：值班运维 -> Big Data Architect -> 研发负责人（P0 同步 CEO/CTO）。

### 6.2 排队时长劣化告警（Queue Wait Degrade）

- 指标：`Q_WAIT_P95`。
- 触发（与既有 SLA 策略一致）：
  - P2：`prod-realtime` P95 > 20s 持续 5 分钟。
  - P1：`prod-realtime` P95 > 30s 持续 5 分钟。
  - P0：`prod-realtime` P95 > 45s 持续 10 分钟。
- 收敛：连续 3 个窗口恢复至 P2 以下。
- 升级路径：值班运维 5 分钟确认 -> Big Data Architect 15 分钟给调度策略 -> 研发负责人 30 分钟决定降级/切流。

### 6.3 抢占误伤率告警（Preemption Collateral）

- 指标：`Q_PREEMPT_COLLATERAL`。
- 触发：
  - P2：15 分钟窗口 > 0.5%。
  - P1：15 分钟窗口 > 1.0%。
  - P0：15 分钟窗口 > 2.0%。
- 收敛：连续 2 个窗口低于 0.5%，且无新增关键失败作业。
- 升级路径：Big Data Architect 主导策略收敛 -> 运维执行回滚/限流 -> QA 启动关键链路回归。

### 6.4 GPU 成本偏差告警（GPU Cost Deviation）

- 指标：`Q_GPU_COST_DEV`。
- 触发：
  - P2：单日偏差 > 15%。
  - P1：单日偏差 > 20%。
  - P0：单日偏差 > 30%。
- 收敛：连续 2 日偏差回到 ±10% 内。
- 升级路径：运维核对资源借用与回收 -> Big Data Architect 调整调度窗口建议 -> 财务/业务负责人确认预算修正。

## 7. 运维与 QA 协作边界（RACI）

| 事项 | Big Data Architect | 运维（SRE/平台） | QA | 研发负责人 |
|---|---|---|---|---|
| 指标接入与数据管道落地 | A（定义口径） | R（接入实现） | C | C |
| 阈值变更审批与发布 | A/R（审批） | R（执行发布） | C | C |
| 告警路由与升级维护 | C | A/R | C | C |
| 回归验证与演练 | C | C | A/R | C |
| 月度复盘组织与改进项 | A | R（提供运行数据） | R（质量报告） | C |

> 说明：R=Responsible，A=Accountable，C=Consulted。

## 8. Rollout 计划

### 8.1 第一阶段（T+0 ~ T+7，手工落地）

1. 运维在 Grafana 建 L0/L1 看板，先接入 `Q_CAP_UTIL`、`Q_WAIT_P95`、`Q_PENDING_DEPTH`。
2. Big Data Architect 提供阈值配置表（P2/P1/P0）并进行双人复核。
3. QA 设计 4 类告警的回归用例（容量逼近、排队劣化、误伤率、成本偏差）。
4. 值班流程按手工 runbook 执行，先验证升级链路时效。

### 8.2 第二阶段（T+8 ~ T+21，半自动化）

1. 运维接入 `Q_PREEMPT_COLLATERAL` 与 `Q_GPU_COST_DEV` 自动采集。
2. 增加告警自动收敛判定（去抖/静默窗口），减少重复告警。
3. QA 接入 smoke 回归流水线，每日自动验证关键阈值触发路径。

### 8.3 第三阶段（T+22 ~ T+45，全自动化）

1. 建立“阈值变更 -> 回归验证 -> 灰度发布 -> 自动回滚建议”闭环。
2. 打通月度复盘自动报表（容量/SLA/误伤/成本四维）。
3. 引入趋势漂移自动识别并生成治理 backlog。

## 9. 建议 CEO/运维继续拆分的 Backlog

1. `BD-QUEUE-01-A`：Grafana L0/L1 模板固化与跨环境复用。
2. `BD-QUEUE-01-B`：调度事件总线标准化（抢占、回收、失败码）。
3. `BD-QUEUE-01-C`：告警收敛引擎（去抖、静默、关联抑制）。
4. `BD-QUEUE-01-D`：QA 回归场景自动注入器（模拟峰值/抢占误伤）。
5. `BD-QUEUE-01-E`：月度复盘自动报告与异常归因助手。

## 10. 未决风险

1. 作业标签治理不完整会影响 `Q_PREEMPT_COLLATERAL` 准确性。
2. 成本账单延迟（T+1/T+2）会导致 GPU 偏差告警时效滞后。
3. 多引擎（Spark/Flink/Hive）失败码不统一，可能造成告警漏判。
4. 夜间借用策略与临时扩容动作若未留痕，会干扰趋势判断结果。

## 11. 下一步实施建议

1. **运维接手**：优先完成第一阶段看板与告警通道接入。
2. **QA 接手**：同步落地四类告警回归用例并在一周内完成首轮演练。
3. **架构持续负责**：每周审查阈值命中与误报率，必要时提出修订建议（单独出影响分析）。
