# EAP-0005 Volcano 与大数据融合落地文档

日期：2026-03-26  
责任人：Cloud Native Architect  
状态：进行中（待招聘大数据架构师与评审）

## 1. 任务背景与目标

原始任务包含两部分：

1. 招聘一名大数据架构师。
2. 调研 Volcano 与大数据体系结合方式，并沉淀详细落地文档与架构图。

本交付聚焦第 2 项并为第 1 项提供技术输入，目标是形成可执行、可评审、可拆分实施的落地方案。

## 2. 结论摘要

1. Volcano 可以作为 Kubernetes 上大数据批任务与 AI 训练任务的统一调度底座，核心价值在于组调度（Gang）、队列治理（Queue）与资源回收/抢占策略。
2. 建议采用“统一调度底座 + 引擎分层适配”的方式，支持 Spark、Flink、Ray 三类主要工作负载逐步接入。
3. 落地需要同步建设四条能力线：
   - 调度策略线（Queue/PodGroup/优先级）
   - 平台治理线（命名、配额、租户隔离）
   - 可观测线（排队、等待、失败重试、资源利用）
   - 交付标准线（模板、发布门禁、回滚演练）
4. 在未补齐大数据架构师岗位前，建议先完成 P0/P1 的平台基础工作，不阻塞底座能力建设。

## 3. 目标架构（分层）

### 3.1 接入与数据层

1. 数据来源：业务库 CDC、日志、对象存储、消息总线。
2. 存储分层：
   - 明细层：对象存储（S3/MinIO）或 HDFS
   - 计算加速层：缓存/列式格式（Parquet/ORC）
   - 元数据层：Hive Metastore / Catalog 服务
3. 对外通过统一数据目录进行表与权限治理。

### 3.2 计算引擎层

1. Spark-on-K8s：离线批处理、ETL、特征加工。
2. Flink-on-K8s：流批一体、实时聚合与实时指标。
3. Ray-on-K8s：分布式 Python 任务、AI 推理与训练前后处理。

### 3.3 调度与资源治理层（Volcano）

1. Queue 负责租户和业务域资源边界。
2. PodGroup 负责强关联任务的整组可运行性。
3. 动作链（enqueue/allocate/preempt/reclaim/backfill）负责调度决策。
4. PriorityClass 与 Queue 共同定义 SLA 优先级。

### 3.4 平台与可观测层

1. 指标：排队时长、调度成功率、作业失败率、重试次数、GPU/CPU 利用率。
2. 日志与事件：调度失败原因、抢占事件、回收事件、驱逐事件。
3. 成本治理：按队列和业务域统计资源消耗。

## 4. 与大数据引擎的集成策略

### 4.1 Spark + Volcano

1. Spark Driver/Executor 使用 `schedulerName: volcano`。
2. 为同一应用的 Driver/Executor 配置 PodGroup，避免 Executor 资源不足导致作业长时间无效等待。
3. 不同业务域绑定不同 Queue，避免共享集群互相挤压。

### 4.2 Flink + Volcano

1. Flink JobManager/TaskManager 采用统一 Queue 策略。
2. 对强实时任务设高优先级队列，对准实时任务设中优先级队列。
3. 配置容量上限与回收策略，防止实时业务在高峰被离线作业挤占。

### 4.3 Ray + Volcano

1. Ray Head/Worker 通过 PodGroup 约束最小可用 Worker 数。
2. 对训练任务区分“可中断”和“不可中断”队列。
3. 与模型流水线配合时，按阶段拆分 Queue 权重，控制端到端时延。

## 5. 多租户队列设计建议

建议初期采用 4 队列模型：

1. `prod-realtime`：生产实时，最高优先级，严格保底。
2. `prod-batch`：生产离线，中高优先级，可控回收。
3. `ai-train`：训练任务，中优先级，允许窗口性借用。
4. `dev-sandbox`：开发测试，低优先级，可被抢占。

治理原则：

1. 关键队列必须有 `guarantee`。
2. 非关键队列允许 `reclaimable`。
3. 生产与开发不共用高优先级队列。
4. 所有提交必须显式声明 Queue 与资源请求。

## 6. 测试与验收矩阵

> 目标：将验收标准从“方向性描述”改为“可量化、可留证据、可复盘”。

| 优先级 | 测试项 | 通过标准 | 证据产物 |
|---|---|---|---|
| P0 | Volcano 基础可用性（Queue/PodGroup/调度链） | 连续 3 天调度成功率 >= 99.5%，无 P0 故障 | `kubectl` 事件导出、调度成功率看板截图、发布记录 |
| P0 | 生产实时队列保底能力（`prod-realtime`） | 高峰期 SLA 作业排队 P95 <= 30s；保底资源始终满足 | 队列利用率报表、SLA 作业运行日志、告警闭环记录 |
| P1 | Spark 批处理试点链路 | 试点链路连续 7 天成功率 >= 99%，失败可在 30 分钟内定位根因 | 作业成功率报表、失败案例 RCA、运行参数快照 |
| P1 | Flink 实时链路治理 | 峰值流量下端到端延迟抖动 < 20%，无长时间积压 | 延迟指标曲线、Backpressure 报告、扩缩容记录 |
| P2 | Ray 训练/推理混部 | 可中断训练任务被回收后可自动恢复；关键推理任务无抢占中断 | 任务恢复记录、回收事件日志、SLA 对比报表 |
| P2 | 抢占与回收策略灰度 | 抢占触发后关键业务无误伤，回收命中率 >= 90% | 抢占事件清单、影响面评估、灰度开关记录 |
| P3 | 故障演练（调度器异常/节点异常） | 30 分钟内完成故障切换或回滚，业务恢复时间满足 RTO | 演练剧本、执行记录、复盘文档 |
| P3 | 容量与成本季度复盘 | 队列资源利用率与成本偏差在预算阈值内（<= 10%） | 季度容量报告、成本分析、优化项跟踪单 |

## 7. 队列参数基线

> 初始值用于首轮生产基线，后续按季度容量评审调整。

| 队列 | 目标负载 | guarantee（保底） | capacity（上限） | 抢占/回收策略 | 变更回滚触发条件 |
|---|---|---|---|---|---|
| `prod-realtime` | 实时核心链路 | CPU 35%，内存 35%，GPU 20% | CPU 60%，内存 60%，GPU 35% | 默认不被回收；仅可对低优先级队列发起抢占 | 连续 10 分钟 SLA 延迟超阈值，立即回滚到上版策略 |
| `prod-batch` | 离线 ETL/报表 | CPU 25%，内存 25%，GPU 10% | CPU 50%，内存 50%，GPU 20% | 可被 `prod-realtime` 抢占；允许夜间窗口借用空闲资源 | 作业失败率连续 2 个周期 > 2%，回滚 |
| `ai-train` | 训练与大模型批任务 | CPU 20%，内存 20%，GPU 40% | CPU 45%，内存 45%，GPU 70% | 可中断任务允许回收；不可中断任务进入专用子队列 | GPU 等待 P95 > 15 分钟且影响线上推理时回滚 |
| `dev-sandbox` | 开发测试 | CPU 5%，内存 5%，GPU 0% | CPU 20%，内存 20%，GPU 10% | 始终可被抢占/回收，优先释放 | 工作时段出现生产队列资源饥饿即降配并回滚 |

参数治理约束：

1. 所有队列变更走变更单并附容量评估。
2. `guarantee` 与 `capacity` 同步评审，不允许只调上限不调保底。
3. 每次变更必须定义“观测窗口 + 回滚阈值 + 决策人”。

## 8. 风险触发与升级机制

### 8.1 告警阈值（触发器）

1. 调度排队：`prod-realtime` 排队时长 P95 > 30s（持续 5 分钟）触发 P1 告警。
2. 调度成功率：任一生产队列 15 分钟窗口成功率 < 99% 触发 P1；< 97% 触发 P0。
3. 抢占误伤：高优先级抢占导致关键作业失败率 > 1%（15 分钟窗口）触发 P0。
4. 资源饥饿：`prod-realtime` 实际可用资源低于保底 10% 以上（持续 10 分钟）触发 P0。
5. 成本异常：单日 GPU 成本偏差 > 20% 触发 P2，并启动容量复核。

### 8.2 升级路径

1. 值班平台工程师（5 分钟内确认） -> 云原生架构师（15 分钟内收敛策略） -> 研发负责人（30 分钟内业务决策）。
2. P0 事件需同步 QA 与相关业务 Owner，进入 War Room。
3. 事件结束后 24 小时内提交 RCA 与策略修订项。

### 8.3 回滚决策人

1. 调度策略回滚决策人：Cloud Native Architect。
2. 涉及业务窗口切换的最终确认人：研发负责人（或其授权值班经理）。
3. 生产实时链路的强制回滚条件命中时，允许平台值班先执行技术回滚，再补审批记录。

## 9. 交付门禁与平台规范

1. 作业模板必须显式声明 `schedulerName: volcano`。
2. 禁止未声明 `requests/limits` 的任务进入生产队列。
3. 每个任务必须绑定业务域标签（`biz-domain`）和成本标签（`cost-center`）。
4. 每次策略调整必须附带回滚方案与回放验证记录。

## 10. 分阶段实施计划

### P0（1 周）平台准备

1. 固化版本基线与兼容矩阵。
2. 建立 Queue/PodGroup 命名与权限规范。
3. 提供 Spark/Flink/Ray 模板样例。

### P1（2 周）单业务试点

1. 选择 1 条离线链路接入 Spark + Volcano。
2. 打通指标看板：排队、成功率、失败原因。
3. 输出第一版运维手册与故障定位 SOP。

### P2（2-4 周）多引擎扩展

1. 接入 Flink 实时链路。
2. 接入 Ray 推理或训练前处理链路。
3. 启用分级优先级和容量治理。

### P3（持续）治理优化

1. 引入回收/抢占策略并按业务分层。
2. 建立季度容量评估与成本复盘机制。
3. 形成数据平台 SLA 与调度策略联动机制。

## 11. 风险与缓解

1. 风险：调度策略过于激进导致关键任务抖动。  
   缓解：生产先禁用抢占，灰度开启并设置保护阈值。
2. 风险：多引擎同时接入导致排障复杂。  
   缓解：统一模板、统一标签、统一日志字段。
3. 风险：无大数据架构师导致领域决策偏差。  
   缓解：先完成底座建设，同时并行推进招聘并引入架构评审机制。
