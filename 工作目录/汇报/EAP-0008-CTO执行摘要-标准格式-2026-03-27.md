## [任务理解]

本期目标是把 AI Infra 面试平台从当前 Go 演示主链路，推进到可持续演进的一期云原生架构方案，而不是继续堆单点演示能力。范围包括：总体架构设计、后端 Go + Operator 路线复核、Spark/Flink 与 Volcano 融合方案、平台可视化与 Grafana 观测边界、以及跨角色派工与评审门禁。硬约束已冻结：后端必须用 Go，控制面优先 CRD/Controller，中间件优先 PostgreSQL/Redis/Kafka，流批必须接 Spark/Flink + Volcano，禁止 YARN。非目标是本周内一次性交付完整生产级多租户数据平台，也不把高频业务状态机下沉进 Operator。

关键假设：

1. 一期以单 Kubernetes 集群为主运行底座；
2. 当前 Go 服务仍可作为后续正式架构的演示入口，但必须向模块化单体 + Operator 控制面演进；
3. D1 目标是“冻结边界与字典”，不是交付全部实现。

## [总体架构]

### 应用层

1. 候选人前端：面试启动、答题、提交、结果摘要。
2. 平台前端：平台总览、运行列表、告警入口、Grafana 深链入口。
3. Go Interview API：会话编排、题目/答案/评测编排、结果查询。
4. Platform API：平台态聚合接口（overview/runs/alerts）。

### 数据层

1. PostgreSQL：会话、答案、评测、平台配置、outbox。
2. Redis：幂等预占、缓存、短期状态加速。
3. Kafka：业务事件、评测命令、运行事件、死信事件。
4. MinIO / Iceberg / Hive Metastore：承接批流加工结果。
5. Prometheus + Grafana + Loki：指标、日志、告警、深链观测。

### 控制面

1. `InterviewPlatform`：管理 API、配置、依赖连接、版本与运行状态。
2. `BatchPipeline`：管理 Spark 作业模板、数据集与 Volcano 队列映射。
3. `StreamPipeline`：管理 Flink 作业模板、Kafka Topic、Checkpoint 与 SLA。
4. Controller 仅负责资源编排、状态收敛、回滚与可观测接入，不持有高频业务状态机。

### 运行面

1. Kubernetes：统一运行底座。
2. Volcano：统一调度与队列治理。
3. Spark on Kubernetes：离线 ETL、报表、批任务。
4. Flink on Kubernetes：实时流处理、实时指标、准实时分析。
